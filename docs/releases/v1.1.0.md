# Lunie v1.1.0

## Overview

Lunie v1.1.0 is a usability-focused release centered on one major change: operators no longer need to copy raw internal IDs for day-to-day CLI and TUI workflows.

This release introduces durable human-friendly identifiers across the stack. Workflows now have generated keys, triggers have workflow-scoped keys, runs have per-workflow run numbers, and the CLI/TUI/docs have been updated around those stable references. The result is a much cleaner operator experience without giving up explicitness or inspectability.

## Highlights

- Durable workflow keys, trigger keys, and per-workflow run numbers
- Key-based API routes for workflows, triggers, versions, runs, and steps
- CLI and TUI updated to use workflow keys, trigger keys, run numbers, step keys, and secret names instead of raw IDs
- Public webhook URLs now use workflow keys and trigger keys in the path

## Install

### Homebrew

```bash
brew tap gentij/lunie
brew install lunie
```

### AUR

```bash
yay -S lunie-cli-bin
```

### GitHub Release Binary

Download the matching archive from:

- https://github.com/gentij/lunie/releases

Extract `lunie` and put it on your `PATH`.

## Upgrade Notes

- Fresh installs do not require any special action.
- If you were testing an older local pre-release database built before durable identifiers were enforced, the simplest upgrade path is to reset the local database and re-run migrations.
- Existing raw ID-based server routes are still present for compatibility, but the primary operator-facing CLI and TUI flows now use keys and run numbers.
- Public webhook URLs now follow `.../hooks/:workflowKey/:triggerKey/:webhookKey`.
